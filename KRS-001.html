Abstract

This document defines a Kraskaska Repository Specification (abbreviated as KRS). KRSes are documents defining specifications, protocols, standards, or any other information that requires rigid structuring and clear definitions. Besides the KRSes themselves this document also describes the process of creating a KRS, and getting it included into the official Kraskaska Repository.

Note: KRSes are not meant to be taken seriously. KRSes are not RFCs - internet standards published by IETF - even if they may look as such.

1. Introduction

Kraskaska Repository Specification (abbreviated as KRS) is a process of defining standards in a rigid, structured format, as well as the name of the standards themselves. During this process, a new KRS is defined following the structure described in this KRS.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

2. Process outline

Process is, generally, as follows:

  1. A need for a standard appears.
  2. A draft of associated KRS is written.
  3. The KRS board MUST vote on whether the KRS is needed.
  4. If the vote passes, the KRS draft is now under consideration.
  5. The draft SHOULD be revised by interested parties.
  6. Interested parties MAY comment on the latest revision, suggesting changes and implementing them in a new revision. This is repeated, until...
  7. A final revision is made.
  8. The KRS board MUST vote on whether the final revision of KRS is ready for publication, following the established procedures.
  9. If the vote passes, the final revision of KRS is published in the repository, making the KRS official and granting it a permanent KRS index .

The KRS draft MAY get dropped after being put under consideration, if the parties become disinterested in the standardization.

Once the KRS is published it MUST NOT be modified in any way, except for changing metadata, fixing formatting issues and correcting typos.

If the need arises a newer KRS MAY obsolete an older published KRS. If that were to happen the newer KRS SHOULD be prioritised over the older obsoleted KRS.

Instead of obsoleting a newer KRS MAY instead update an existing published KRS. When referencing the KRS the original and all associated updates SHOULD be considered together. If the original KRS is obsoleted all KRSes that update the original MUST be considered obsolete as well.

If the need arises there MAY be an errata published that corrects the mistakes in the contents of the KRS, or that explains vague details that were left out, accidentally or not. The erratas do not have their own index, but they MUST refer to a published KRS permanent index. The erratas, like KRSes, MUST NOT be modified in any way, except for changing metadata, fixing formatting issues and correcting typos.

3. Specification document structure

A KRS document MUST be a valid HTML5 document, containing an html tag with the attribute lang set to en. This section describes the structure of the document.

3.1. Metadata

The document MUST contain a head tag.

The head tag MUST contain a title tag containing the title of the KRS document, which follows the format:

KRS-XXX: T

XXX is the permanent index assigned to the KRS, if any, with leading zeros to pad the index to at least 3 digits. If no permanent index is assigned to the document (due to it still being a draft or pending publication) the XXX MAY be set to the estimated index that would be granted to the KRS after publication, or the XXX MAY be set to the word "PENDING".

T is the title of the standard, and it doesn't follow any semantics.

The head MUST also contain a script tag with attribute type set to application/yaml. This tag contains YAML metadata specifying KRS properties in a machine readable form.

3.2. Document body

The document MUST contain a body tag. In it there MUST be at least a header and a main tag. The body contains the description of the KRS written in formal English language.

3.2.1. Header

The header tag MUST contain an h1 that MUST match the contents of the previously defined title tag.

The header MUST also contain a p tag or a ul tag containing the author or authors of the KRS document.

The header MUST also contain a p tag specifying the SPDX of an OSI-approved license, which applies to the whole document. The KRS board will reject publication of KRSes that do not have an OSI-approved license.

The header tag MAY contain other elements not defined here.
