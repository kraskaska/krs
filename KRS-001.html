Abstract

This document defines a Kraskaska Repository Specification (abbreviated as KRS). KRSes are documents defining specifications, protocols, standards, or any other information that requires rigid structuring and clear definitions. Besides the KRSes themselves this document also describes the process of creating a KRS, and getting it included into the official Kraskaska Repository.

Note: KRSes are not meant to be taken seriously. KRSes are not RFCs - internet standards published by IETF - even if they may look as such.

1. Introduction

Kraskaska Repository Specification (abbreviated as KRS) is a process of defining standards in a rigid, structured format, as well as the name of the standards themselves. During this process, a new KRS is defined following the structure described in this KRS.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

2. Process outline

Process is, generally, as follows:

  1. A need for a standard appears.
  2. A draft of associated KRS is written.
  3. The KRS board MUST vote on whether the KRS is needed.
  4. If the vote passes, the KRS draft is now under consideration.
  5. The draft SHOULD be revised by interested parties.
  6. Interested parties MAY comment on the latest revision, suggesting changes and implementing them in a new revision. This is repeated, until...
  7. A final revision is made.
  8. The KRS board MUST vote on whether the final revision of KRS is ready for publication, following the established procedures.
  9. If the vote passes, the final revision of KRS is published in the repository, making the KRS official and granting it a permanent KRS index .

The KRS draft MAY get dropped after being put under consideration, if the parties become disinterested in the standardization.

Once the KRS is published it MUST NOT be modified in any way, except for changing metadata, fixing formatting issues and correcting typos.

If the need arises a newer KRS MAY obsolete an older published KRS. If that were to happen the newer KRS SHOULD be prioritised over the older obsoleted KRS.

Instead of obsoleting a newer KRS MAY instead update an existing published KRS. When referencing the KRS the original and all associated updates SHOULD be considered together. If the original KRS is obsoleted all KRSes that update the original MUST be considered obsolete as well.

If the need arises there MAY be an errata published that corrects the mistakes in the contents of the KRS, or that explains vague details that were left out, accidentally or not. The erratas do not have their own index, but they MUST refer to a published KRS permanent index. The erratas, like KRSes, MUST NOT be modified in any way, except for changing metadata, fixing formatting issues and correcting typos.

3. Specification document structure

A KRS document MUST be a valid HTML5 document, containing an html tag with the attribute lang set to en. This section describes the structure of the document.

3.1. Metadata

The document MUST contain a head tag.

The head tag MUST contain a title tag containing the title of the KRS document, which follows the format:

KRS-XXX: T

XXX is the permanent index assigned to the KRS, if any, with leading zeros to pad the index to at least 3 digits. If no permanent index is assigned to the document (due to it still being a draft or pending publication) the XXX MAY be set to the estimated index that would be granted to the KRS after publication, or the XXX MAY be set to the word "PENDING".

T is the title of the standard, and it doesn't follow any semantics.

The head MUST also contain a script tag with attribute type set to application/yaml. This tag contains YAML metadata specifying KRS properties in a machine readable form.

3.2. Document body

The document MUST contain a body tag. In it there MUST be at least a header and a main tag. The body contains the information and description of the KRS written in formal English language.

3.2.1. Header

The header tag MUST contain an h1 that MUST match the contents of the previously defined title tag.

The header MUST also contain a p tag or a ul tag containing the author or authors of the KRS document.

The header MUST also contain a p tag specifying the SPDX identifier of an OSI-approved license, which applies to the whole document. The KRS board will reject publication of KRSes that do not have an OSI-approved license.

The header MUST also contain a p tag specifying the status of the document. The status may be one of the following:

  • Draft - the document is a draft, and hasn't yet been voted upon to be considered for inclusion in KRS repository.
  • Under consideration - the document has passed a vote on being included in the KRS repository, and is now under active revisioning by interested parties.
  • Published - the document was finalized and passed the vote for publication, and is now published in the repository.
  • Dropped - the document was dropped because of a lost interest in standardization and a pass of vote for dropping.

Documents that are in repository MUST have a status of published

The header MAY contain a p tag specifying a KRS document that this KRS document updates.

The header MAY contain a p tag specifying KRS documents that this KRS document obsoletes.

The header tag MAY contain other elements not defined here.

3.2.2. Description

The main tag contains the description of the KRS document. It consists of multiple sections, that MUST contain a heading (h2, h3, h4, h5, h6) and MAY contain a body, and subsections that follow the same rules.

3.2.2.1. "Abstract" section

The description of KRS document MUST contain an abstract section inside a header tag, instead of regular section tag. The heading MUST be "Abstract". This section MUST contain a short summary of what the KRS document describes. The contents of this section MAY be inlined by a KRS viewer (program that parses and displays the KRSes).

3.2.3. Footer

The body MAY contain a footer tag preceeded by an hr tag. The footer MAY contain the contact information of the authors of the KRS document.

4. KRS Document YAML Metadata

This section describes the KRS Document YAML metadata that is included in the head > script type="application/yaml" element of the document.

4.1. schema

The YAML root key schema MUST contain a reference to the schema KRS document that defines the structure of the KRS document. It MUST follow format of KRS-XXX, where XXX MUST be the permanent index assigned to the published KRS, if any, with leading zeros to pad the index to at least 3 digits, that defines the structure of a KRS document as well as its metadata.

4.2. index

The YAML root key index MUST contain the assigned permanent index of the KRS, or the string pending if it's not available yet. The value MUST be an unadorned integer with no leading zeroes.

4.3. title

The YAML root key title MUST contain the title of the document as defined in section 3.1. of this document. Title in YAML and title in head MUST be equivalent.

4.4. status

The YAML root key status MUST contain the publication status of the KRS document. The publication status MUST be one of the following:

  • draft - the document is a draft, and hasn't yet been voted upon to be considered for inclusion in KRS repository.
  • under-consideration - the document has passed a vote on being included in the KRS repository, and is now under active revisioning by interested parties.
  • published - the document was finalized and passed the vote for publication, and is now published in the repository.
  • dropped - the document was dropped because of a lost interest in standardization and a pass of vote for dropping.

Documents that are in repository MUST have a status of published

Contents of this key and the status in HTML document body > header MUST be equivalent.

4.5. authors

The YAML root key authors MUST contain an array of authors that worked on this KRS. Contents of this key and the list of authors in HTML document body > header MUST be equivalent.

4.6. license

The YAML root key license MUST contain the SPDX identifier of an OSI-approved license, which applies to the whole document. The KRS board will reject publication of KRSes that do not have an OSI-approved license. Contents of this key and the license in HTML document body > header MUST be equivalent.

4.6. updates

The OPTIONAL YAML root key updates MAY contain the assigned permanent index of a KRS that this document updates. Contents of this key and paragraph specifying the updated KRS in HTML document body > header MUST be equivalent.

4.6. obsoletes

The OPTIONAL YAML root key obsoletes MAY contain an array of the assigned permanent indicies of a KRS that this document obsoletes. Contents of this key and paragraph specifying the obsoleted KRSes in HTML document body > header MUST be equivalent.

4.7. Other tags

YAML MAY contain other root keys not defined here, but they MUST be prefixed.

4.7.1. KRS keys

Other KRSes MAY define extra YAML root keys prefixed with krs-xxx-, where xxx is assigned permanent index of a KRS that defines the semantics and contents of the key.

4.7.2. Vendor-specific keys

Root keys that are not defined in KRSes MAY be included in YAML if they are prefixed with x-. The semantics of such keys are undefined.

5. The KRS board

The KRS board performs tasks in maintaining the KRS repository as well as performing voting for the KRS standards to be included in the repository.

Currently, KRS Special Interest Group (KRS-SIG) assumes the position of the KRS board.

KRS-SIG consists of members interested in advancement of the KRS repository. The members perform the duties of KRS board, as well as any other action within KRS-SIG by initiating voting, and the actions are carried out depending on the results of the voting.

One of the members of KRS-SIG has a role of a leader. A leader may only be chosen by previous leader, or, in the event of an emergency, by voting if asking the opinion of the previous leader is no longer physically possible due to circumstances beyond the control of the group. The leader may perform any action and any duties of the group WITHOUT initiating voting, and may also override the voting results.

6. Canonical repository location and board membership

Currently, the canonical location for the KRS repository is GitHub repository located at kraskaska/krs (https://github.com/kraskaska/krs).

KRS-SIG members currently consist of the members of the KRS-SIG team in the GitHub organization kraskaska (https://github.com/kraskaska).

Currently, the leader of KRS-SIG is Kraskaska.
